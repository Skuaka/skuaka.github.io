mp3文件格式

MP3 文件结构

MP3文件大体分为三部分:TAG_V2(ID3V2)，Frame, TAG_V1(ID3V1)

ID3V2 包含了作者，作曲，专辑等信息，长度不固定，扩展了 ID3V1 的信息量。
Frame . . . Frame 一系列的帧，个数由文件大小和帧长决定 每个 FRAME 的长度可能不固定，也可能固定，由位率 bitrate 决定 每个 FRAME 又分为帧头和数据实体两部分 帧头记录了 mp3 的位率，采样率，版本等信息，每个帧之间相互独立
ID3V1 包含了作者，作曲，专辑等信息，长度为 128BYTE。

ID3V2

ID3V2 到现在一共有4 个版本，但流行的播放软件一般只支持第3 版， 既ID3v2.3。

由于ID3V1 记录在MP3 文件的末尾，ID3V2就只好记录在MP3 文件的首部了(如果有一天发布ID3V3，真不知道该记录在哪里)。也正是由于这个原因，对ID3V2 的操作比ID3V1 要慢。而且ID3V2 结构比ID3V1 的结构要复杂得多，但比前者全面且可以伸缩和扩展。

下面就介绍一下ID3V2.3：

每个ID3V2.3 的标签都一个标签头和若干个标签帧或一个扩展标签头组成。关于曲目的信息如标题、作者等都存放在不同的标签帧中，扩展标签头和标签帧并不是必要的，但每个标签至少要有一个标签帧。标签头和标签帧一起顺序存放在MP3 文件的首部。

1、标签头

在文件的首部顺序记录10个字节的ID3V2.3 的头部。数据结构如下：

char Header[3]; /必须为”ID3”否则认为标签不存在/

char Ver; /版本号ID3V2.3 就记录3/

char Revision; /副版本号此版本记录为0/

char Flag; /存放标志的字节，这个版本只定义了三位，稍后详细解说/

char Size[4]; /标签大小，包括标签头的10个字节和所有的标签帧的大小/

1）标志字节

标志字节一般为0，定义如下： abc00000

a – 表示是否使用Unsynchronisation(一般不设置)

b – 表示是否有扩展头部(一般不设置)

c – 表示是否为测试标签(一般不设置)

第6个字节：存放标志的字节，只定义了三位，这里值为0

2）标签大小

一共四个字节，但每个字节只用7位，最高位不使用恒为0。所以格式如下

0xxxxxxx 0xxxxxxx 0xxxxxxx 0xxxxxxx

计算大小时要将0 去掉，得到一个28 位的二进制数，就是标签大小，计算公式如下：

total_size = (Size[0]&0x7F)0x200000+ (Size[1]&0x7F)0x4000 + (Size[2]&0x7F)*0x80 +(Size[3]&0x7F)

注意：这里的帧大小，并不包含帧头的10个字节，只表示帧内容的大小.

2、标签帧

每个标签帧都有一个10个字节的帧头和至少一个字节的不固定长度的内容组成。它们也是顺序存放在文件中，和标签头和其他的标签帧也没有特殊的字符分隔。得到一个完整的帧的内容只有从帧头中的到内容大小后才能读出，读取时要注意大小，不要将其他帧的内容或帧头读入。

帧头的定义如下：

char ID[4]; /用四个字符标识一个帧，说明其内容，稍后有常用的标识对照表/

char Size[4]; /帧内容的大小，不包括帧头，不得小于1/

char Flags[2]; /存放标志，只定义了6 位/

1）帧标识

用四个字符标识一个帧，说明一个帧的内容含义，常用的对照如下：

TIT2=标题表示内容为这首歌的标题，下同

TPE1=作者

TALB=专集

TRCK=音轨格式：N/M 其中N 为专集中的第N 首，M为专集中共M 首，N和M 为ASCII 码表示的数字

TYER=年代是用ASCII 码表示的数字

TCON=类型直接用字符串表示

COMM=备注格式：”eng\0备注内容”，其中eng 表示备注所使用的自然语言

2）大小

这个可没有标签头的算法那么麻烦，每个字节的8 位全用，格式如下

xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx

算法如下：

int FSize = Size[0]0x100000000 + Size[1]0x10000+ Size[2]*0x100 + Size[3];

注意：这里的帧大小，并不包含帧头的10个字节，只表示帧内容的大小

3）标志

只定义了6 位，另外的10 位为0，但大部分的情况下16 位都为0 就可以了。格式如下：abc00000ijk00000

a – 标签保护标志，设置时认为此帧作废

b – 文件保护标志，设置时认为此帧作废

c – 只读标志，设置时认为此帧不能修改(但我没有找到一个软件理会这个标志)

i – 压缩标志，设置时一个字节存放两个BCD 码表示数字

j – 加密标志(没有见过哪个MP3 文件的标签用了加密)

k – 组标志，设置时说明此帧和其他的某帧是一组

FRAME

每个 FRAME 都有一个帧头 FRAMEHEADER，长度是 4BYTE(32bit),帧头后面可能有两个字节的 CRC 校

验，这两个字节的是否存在决定于 FRAMEHEADER 信息的第 16bit，为 0 则帧头后面无校验，为 1 则有校验, 校验值长度为 2 个字节，紧跟在 FRAMEHEADER 后面，接着就是帧的实体数据了，格式如下:

FRAMEHEADER CRC(free) MAIN_DATA
4 BYTE 0 OR 2 BYTE 长度由帧头计算得出

1）计算帧长度

我们首先区分两个术语：帧大小和帧长度。帧大小即每帧采样数表示一帧中采样的个数，这是恒定值。其值如下表所示

LAYER MPEG 1 MPEG 2(LSF) MPEG 2.5(LSF)
Layer 1 384 384 384
Layer 2 1152 1152 1152
Layer 3 1152 576 576

帧长度是压缩时每一帧的长度，包括帧头。它将填充的空位也计算在内。LayerI的一个空位长4字节，LayerII和LayerIII的空位是1字节。当读取MPEG文件时必须计算该值以便找到相邻的帧。注意：因为有填充和比特率变换，帧长度可能变化。

从头中读取比特率，采样频率和填充的值后可以进行计算，

LyaerI使用公式：

帧长度（字节） = (( 每帧采样数/ 8 * 比特率 ) / 采样频率 ) + 填充 * 4

LyerII和LyaerIII使用公式：

帧长度（字节）= (( 每帧采样数/ 8 * 比特率 ) / 采样频率 ) + 填充

例：LayerIII 比特率 128000，采样频率 44100，填充0 =〉帧大小 417字节。

(1152 / 8 * 128K)/44.1K = 417 (字节)

2）每帧的持续时间

每帧持续时间(毫秒) = 每帧采样数 / 采样频率 * 1000

3）CRC校验

如果帧头的校验位为0，则帧头后就有一个16位的CRC值，这个值是big-endian的值，把这个值和该帧通过计算得出的CRC值进行比较就可以得知该帧是否有效。

ID3v1

ID3V1标准并不周全，存放的信息少，无法存放歌词，无法录入专辑封面、图片等。ID3V2是一个相当完备的标准，但给编写软件带来困难，虽然赞成此格式的人很多，在软件中绝大多数MP3仍在使用ID3V1标准。ID3v1标签包含艺术家，标题，唱片集，发布年代和流派。另外还有额外的注释空间。位于音频文件的最后固定为128字节。可以读取该文件的最后这128字节获得标签。

Unicode与FFFE

标签帧的内容开头都有 01 ff fe 三个字节。原因是为了判别大端还是小端编码。

在UCS编码中有一个叫做”ZERO WIDTH NO-BREAK SPACE”的字符，它的编码是FEFF。而FFFE在UCS中是不存在的字符，所以不应该出现在实际传输中。UCS规范建议我们在传输字节流前，先传输字符”ZERO WIDTH NO-BREAK SPACE”。
这样如果接收者收到FEFF，就表明这个字节流是Big-Endian的；如果收到FFFE，就表明这个字节流是Little-Endian的。因此字符”ZERO WIDTH NO-BREAK SPACE”又被称作BOM。

CCIE bgp

边界网关路由协议

BGP简介

工作类型：路径矢量路由选择协议

基于tcp(三次握手)

三张表：邻居表（peer表） bgp转发表 路由表

BGP版本

BGPv4：IPv4

BGPv4+：IPv6

MP-BGP：协议npls

M-BGP：多播

BGP优缺点

优点

1EGP路由表只能存储千条信息，bgp可上万条。

2丰富的路径属性，13条选路原则，提供了更丰富的路由操控策略。

缺点

1对路由器性能要求较高。

2配置程序比较复杂。

3一般小网络不建议使用bgp

AS 和 邻居类型

bgp AS 一台路由器只能属于一个

公网AS1-65411 65412-65535私有AS

iBGP：处于同一AS内的BGP路由器所建立的BGP对等体关系

eBGP：不同AS内的BGP路由器建立的…

###

BGP消息

open：建立成功，会互相发送open信息(类似hello)协商邻居关系的各项参数，等待对方确认。

keeplive：维持对等体(类似邻居)关系的建立，60s发一次，180s后peer关系结束

update：不会周期性更新，只会增量更新或触发更新。

Notification：当BGP检测到错误状态时，就向对等体发出Notification消息，之后BGP连接会立即中断。

peer关系建立过程

IDLE（空闲状态）：本地路由向外发送tcp(端口179)请求，等回复

connect状态：tcp建立

active：建立不成功，再次请求tcp连接的建立（回到IDLE）。

open sent：建立成功，发送open消息

openconfirm：收到对方的信息进行参数对比，参数一致则建立完成。

established：连接建立。。update。。keepalive。。

bgp的防环机制

iBGP-水平分隔原则

BGP对等体路由器收到iBGP路由，不会发送给其他iBGP对等体路由器。

但是纯水平拓扑上该原则会导致路由不被传递学习。两个解决方案

全互联拓扑解决方案：所有路由器间都建立iBGP关系。（垃圾方案）

路由反射器解决方案：

eBGP

基于AS PATH信息的检测。接收到来自于eBGP peer的路由更新，更新所带AS-PATH 信息包含了当前设备AS号，则该路由更新忽略。

特殊拓扑下路由反射器的防环机制

originate id：起源id。路由器将设备的RID设置为起源ID。起源ID和当前设备RID一样，则路由更新不学习。

cluster id：簇id。管理员在路由反射器上设置，路由更新携带的簇ID一致，则路由更新不学习。

同步与下一跳判断

路由更新来自于iBGP peer时才考虑同步和下一跳。

同步机制

对于当前启用BGP的设备，路由更新对应的网络底层协议也必须能学习到，这才叫同步。同步是为了避免路由黑洞。

现实中同步被关闭，路由黑洞问题无法避免，但利用MPLS能解决。

bgp路由黑洞

路由更新的传递设备也就是数据传输的沿途设备，并没有参与BGP ，学习该路由，所以数据抵达后，只能丢包。

解决方法：

1重分发(不建议，会增大路由表)

2全互联的网络拓扑(两两之间建立ibgp对等体)

下一跳判断

来自于iBGP peer的路由更新，如果下一跳不可达，路由更新不学习。

路由源于AS外，通告者是eBGP peer，下一跳就是eBGP peer地址。

路由源于AS外，通告者是iBGP peer，下一跳依旧是eBGP peer地址。此时下一跳可能不可达。

路由源于iBGP peer，通告者是iBGP peer，下一跳就是iBGP peer地址。

BGP的路由更新属性

  1. 公认必选

    NEXT-HOP：下一跳

    AS-PATH：AS路径信息

    Originate code：起源代码

    ​ 描述了目标网络进入BGP的方式（I E ?）

    ​ I ：目标网络源于底层协议通过network进入BGP

    ​ E：EGP共享的目标网络信息

    ​ ?：外部路由以重分发的方式进入BGP

  2. 公认任选（自选）

    LP：本地优先

    ​ 路由更新在IBGP peer间传递时，LP值才出现，且为100

    ​ 作用：影响iBGP peer 出站路径的选择，LP参数越大越优。

    automatic aggregate：原子聚合

    ​ BGP发生了自动聚合或手动聚合时，聚合后网络携带该属性。

    ​ 原子聚合属性用于提醒更新接受者，路由更新目标网络可能丢失了部分原有属性。

  3. 任选可传递

    Aggregated：被聚合属性——聚合后的网络携带的属性，表明该网络是聚合来的。

    Community：团体属性 是携带特定策略的标记

    ​ 公认团体属性：

    ​ Internet：所有的BGP路由默认携带该属性，该属性对应的策略是，共享给所有的BGP peer。

    ​ no-export：不导出，携带该属性的路由更新不共享给eBGP peer。

    ​ no-advertise：不通告，携带该属性的路由更新不共享给任何BGP peer。

    ​ local-as：本地as，携带该属性的路由更新不共享给联盟内其他eBGP peer。

    ​ 自定义团体属性：

    ​ AA:NN

    ​ AS号：标记值

  4. 任选不可传递（属性保留，但值为空）

    MED：多出口距离

    ​ eBGP间路由共享携带该属性，MED值用于影响eBGP peer入站路径选择。MED值默认为0，越小越优。

BGP选路原则

选路原则决定了BGP按什么标准进行路由更新属性的比较

  1. 优选拥有最高权重值的路径 weight

    用于影响当前路由器出站路径选择

    默认32768，越大越优

    不同类型的对等体出现，需要影响路径抉择，权重更优。

  2. 优选LP大的

  3. 优选起源于本地的路径

    next-hop=0.0.0.0

  4. 优选拥有最短AS-PATH 路径信息的

  5. 比较Origin属性<IGP优先于EGP优先于Incomplete>

  6. 优选拥有最佳MED值的

  7. eBGP peer 的路径优于 iBGP peer的路径

  8. 优选离下一跳更近的路径

  9. 针对eBGP peer的路由，通过命令行允许多条路径，成为最佳进入路由表

  10. 优选最早被学习的路由，只针对 eBGP peer

  11. 优选拥有最小RID的BGP peer提供的路由

  12. 优选拥有最短cluster-id 的路径

  13. 优选拥有最低peer地址的邻居提供的路径

针对外部路由，auto-summary 开启，外部路由自动汇总，否则外部路由就明细进入。

IGMP 互联网组管理协议

路由器及终端运行，帮助路由器发现直连子网中的多播接收者。

组（group）：所有对同一多播流感兴趣的成员组成了多播组。

组成员（group member）：组中的多播流接收者。

版本：IGMPv1、IGMPv2、IGMPv3

IGMPv1

  • 报告消息（report）：始发于终端，用于通告子网中的路由器存在特定组组成员。

    发送终端的IP 和MAC以及组信息。

  • 查询消息（request）：始发于路由器，用于查询直连子网中是否存在组成员。

    通常是路由器接收到多播流后触发。

一旦发现组成员，路由器就创造一个180s 的老化计时器，该计时器通过 report 消息进行维护。

IGMPv1的缺陷

  1. 响应时间慢

    组中没有成员时，路由器需要 180s 才能发现

  2. 存在多查询者

  3. 存在多转发者

IGMPv2

  • 报告消息（report）：和 v1 相同

  • 查询消息（request）：和 v1 相同

  • 离开消息（leave）：始发于终端，终端PC 不再是组成员时发送，用于通告路由器有组成员离开。

    收到离开消息时，立刻发送SQ，如下。

  • 特定查询消息（SQ）：始发于路由器，发送出去后设定的老化时间为 3s 。如果 3s 内收到 report ，说明发送离开消息的组成员并不是组中最后一个人；如果收不到任何消息，那么说明组成员不存在。

IGMPv2 查询者竞选机制：多个查询者通过比较查询消息源 ip 来确认谁最终负责查询，源 ip 越小越优。

多转发者缺陷 IGMPv2 无法克服。

多播 IP 和多播 MAC 都不是用来定位接收者所在的网络和物理位置。

它们用于标识服务和应用。

多播路由器协议

是运行在路由器之间，负责建立多播路由路径，用于传递多播数据。

多播路由协议分类

  • 距离矢量：DVMRP 距离矢量多播路由协议（淘汰了）
  • 链路状态：MOSPF 多播的OSPF（非主流）
  • 距离向量：CBT 基于核心的树（非主流）
  • 独立于其他：PIM 协议独立多播（Cisco、主流）

PIM 的两个版本

PIMv1：用的是 IGMPv1，但是现在也用 IGMPv2

PIMv2：用的是 IGMPv2

PIM 的三种模式

  • 密集模式（dense mode）

    设备间带宽资源紧张，子网成员大部分是组成员的时候，建议使用密集模式。

    密集模式默认采用 IGMPv1。

  • 稀疏模式（sparse mode）

    设备间带宽资源依旧紧张，子网成员只有少数是组成员，建议使用稀疏模式。

    稀疏模式采用 IGMPv2。

  • 混合模式（sparse-dense mode）

    该模式下路由器先启用稀疏模式，稀疏模式失效后再用密集模式。

PIM 的密集模式详解

消息

  • 加入消息（join）：下游路由器通过加入消息将直连网络加入多播转发路径。
  • 修剪消息（prune）：下游路由器通过修剪消息，将原本隶属于多播路径的直连网络从多播路径中删除。
  • 接枝消息（graft）：下游路由器通过该消息请求将原本被拆除的多播路径再恢复过来。
  • 接枝确认消息（graft-ack）：上游路由器通过该消息告诉下游，直连路径恢复成多播转发路径。
  • assert消息（assert）：连接子网的路由器通过该消息，竞选出最终的转发者。

密集模式工作过程

扩散、剪枝。

PIM 的特殊机制

  • PIM 密集模式的 DR

    接口优先级+接口ip地址，构成竞选参数，越大越优。

    PIM密集模式早期使用IGMPv1，IGMPv1会出现多查询者问题，于是就利用DR的竞选，来帮助IGMPv1确认谁负责查询。

    PIM密集模式现在使用的是IGMPv2，DR就没用了。

  • PIM 消息中的 assert 作用

    通过竞选选出一个人负责转发数据。

    assert 消息，携带了三个竞选参数：

    • 当前路由器启用的底层协议管理距离，越小越优。
    • 当前路由器抵达多播源底层路由协议的开销，越小越优。
    • 当前路由器连接子网的接口 ip，越大越优。

PIM 密集模式的配置流程

  • 步骤1：确认参与多播的设备及接口

    所有需要接收多播流的接口和发送多播流的接口及连接组成员的接口，都要参与多播路由协议的启用。

  • 步骤2：路由器接口模拟成多播组成员

    注意：该命令可以反复录入，一个接口扮演多个组成员

  • 步骤3：路由器开启多播路由功能，接口启用多播密集模式

  • 步骤4：校验查看

    校验路由器是否发现了组成员：show ip igmp group

    校验路由器是否形成了PIM 邻居关系：show ip pim neighbor

    校验路由器多播路由表：show ip mroute

多播路由表

父条目与子条目

  • 父条目

    父条目起提醒注释的作用，告知当前路由器和管理员，路由器可能需要为某个多播组传递流量。

    路由器收到 report 消息便会产生父条目，路由器收到多播流也会产生父条目。

  • 子条目

    子条目是实际负责多播流量传递工作。

    子条目是路由器根据计算得来。

多播路由条目的样式

（多播源，组地址） 存活计时器/老化计时器

Incoming-interface（上游接口）：接口名字 状态/模式

Outgoing-interface（上游接口）：接口名字 状态/模式

父条目多播源部分几乎都是*号，代表源未知。

子条目多播源部分一定是某个 IPv4 单播地址。

多播防环机制

RPF：反向路径转发：启用多播的接口默认开启，用于避免环路的出现。

所有接收多播流的接口，必须通过 RPF 检测，才可以成为有效的上游接口。

当前接口必须是抵达多播源下一跳地址的本地出口，才通过检测。

用来查看rpf检测是否通过：show ip rpf <intf-id>

多播树

多播树其实就是多播转发数据的路径。

密集模式的树

最短路径树，也称源树。

当前路由器以多播源为根，计算抵达多播源的最短路径，每个路径对应到多播路由表中的一个条目。

源树缺陷
  • 容易产生大量的多播子条目。
  • 树构建的早期多播流被扩散到了不必要的设备上

PIM 的稀疏模式详解

消息

join、prune、graft、graft-ack、assert（和密集模式相同）

注册消息（register）

注册停止消息（register-stop）

稀疏模式中的集合点：RP

RP 是管理员指定的，是多股多播流的交汇处。

稀疏模式的工作原理

上游注册，下游加入，RP 整合。

！！注意

RP 以下的设备，以 RP 为根，计算最短路径。

RP 自身以源为根，计算最短路径。

稀疏模式是一个双树结构，RP 和源之间维护的是源树，下游设备与 RP 之间维护的是共享树。

共享树切换成源树

稀疏模式下，每台支持多播路由协议的路由器在接收超过4个字节的多播流后，就会尝试建立源树。

RP 的获取方式

      C-RP：RP候选者

      RP-Agent：RP代理、C-RP 信息收集，裁决，分享结果

      RP：RP

    2. BSR：boot strap router：自举路由器

      C-BSR：BSR 候选人

      BSR：信息收集，信息分享

      C-RP、RP

      BSR 的竞选过程：BSR 标识越大越优。

      RP

      C-RP 有三个竞选参数：优先级（小优）、hash 长度（长优）、C-RP 标识（大优）

CCIE 跨域的多播

跨域的多播

源和组成员在不同的AS，需要传递多播数据流。

跨域多播的问题和挑战

组成员无法发现源、源无法发现组成员、多播树无法建立。

MSDP 多播源发现协议

MSDP 是配置在不同 AS RP 上的，它会建立基于 TCP 的对等体关系，利用自身的 SA（源关联）消息共享”源地址信息、多播组地址信息、RP 信息”给对端 RP。

MSDP 配置流程

步骤1：RP上配置新的 loopback 接口，并且宣告进 BGP。

步骤2：RP上完成 MSDP 的配置

排错手段

ip 路由疑难解析

多播排错

ping 多播组地址 不同

  1. 确认发现组成员

    show ip igmp group

    查看组地址、本地接口、报告者 是否出现

    错点

    • 没有模拟成组成员，缺乏接口下：ip igmp join-group <a.b.c.d>
    • 路由器接口没有加入PIM，缺乏接口下:ip pim <x>
    • 连通性问题

  2. 确认稀疏模式获取 RP 信息

    show ip pim rp mapping 查看 RP 映射，查看 RP 的结果

    show ip pim rp 查看多播组和 RP 的关联

    错点

    • 没有通过静态 RP 指明 RP 地址
    • BSR 的 C-BSR 或 C-RP 配置问题
    • auto-rp 的配置问题
    • 缺乏auto-rp 侦听的配置
    • 设备间缺乏必要的PIM 邻居：show ip pim neighbor
      • 设备缺乏多播路由功能和接口忘记加入PIM
      • 接口忘记激活、错误的安全策略挡住了PIM消息
    • 设备无法正常抵达 RP 地址，所以不显示获取的 RP 消息
      • 路由层面问题、忘记宣告、忘记重分发、错误的过滤

  3. RPF 检测未通过

    • 缺乏必要的更新源路由
    • 忘记宣告、忘记重分发、错误的路由更新过滤

  4. 是否是跨域多播、缺乏必要的配置和连通性

    • 缺乏必要的 MSDP 配置和 MSDP 对等体关系

      show run | sec ip msdp

      show ip msdp summary

      MSDP peer 地址不可达

      MSDP 链接源和起源 ID 设置错误

    • RPF 检测不通过

  5. 错误的安全策略

重分发

将不同路由协议转化

1直连进路由选择协议

redistribute connect metric 1

2其他路由选择协议进rip

router rip

router下 redistribute eigrp 100 metric 1值(默认0)

3其他路由选择协议进eigrp

router eigrp 100

redistribute rip metric 10000带宽 100延迟 255可靠性 1负载 1500MTU值（默认为以上数字，一般不改）

4其他进ospf

router os 1

redistribute eigrp 100进程号 subnets（子集，不加则为主类网络）

默认OE2

将oe2改成oe1:

re ei 100 subnets metric-type 1 (+metric 值）

acl

奇acc 1 permit 172.16.1.0 0.0.254.255

偶acc 1 permit 172.16.0.0 0.0.254.255

将acl抓去到的流量给route-map

route-map +名字 +permit/deny

优先级10(默认优先级)

match ip address +acl编号

route-map也默认deny所有流量

如需放行所有流量 route-map 名字 permit +下一个优先级20(一般为10的叠加)

re rip router-map ccie metric ……
